# Payment Gateway Design Spec
This document is about the key points in the design of the payment gateway service.

# Components

## Bank Adapter
* The bank adapter implements the payment protocol of the bank. 
* The bank adapter manages the configuration of the bank. For example, the endpoint of the bank.
* The request validation logic is implemented as a part of the bank adapter, because different banks may have different validation requirements.
* In production, the bank adapter can be implemented as a plug-in in a different binary. In this way, the adapter and the gateway service can have different release cycles.

## MounteBank Adapter
* The adapter of the bank simulater in this demo. 
* The request validation requirement is defined in validation_spec.md. Each requirement has a number for human review.
* The implementation is generated by AI in `ValidateRequest()`. 
* The test cases is generated by AI in ValidationTest.cs.

## Bank Adapter Factory
* Returns the IBankAdapter instance to the bank name. If we are goint to support multiple banks, we may use this pattern.
* The test code can mock the Adapter and Factory using `Moq`.


# Other Design points
## Payment ID
* The Payment ID is generated immediately on receiving the payment request whenever 
* It keeps unchanged during the process of the payment and will be preserved in DB. 
* It can be used for logging and tracking the payment footprint.

## Authorization Code
* Authorization Code is kept as string, because some bank may use other format.
* Authorization Code cannot be used as Payment ID. In error cases, the Authz Code is not generated but we still need to track the payment.

## Amount data type
* In this example, the amount is Int32. If the caller inputs a value exceeds the max Int32 value, the service will hit Json parsing error.
* We may considering using string data type in protocol and apply a more strict validation on receiving.

## CVV type
* Changed to string. The CVV can have leading zeros.
